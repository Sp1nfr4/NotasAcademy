Herramientas para tomar Notas

Existen muchas herramientas de toma de notas que pueden ser múy útiles en nuestro trabajo; vamos a hablar de algunas de ellas; es importante resaltar que actualmente muchas herraminetas usan el lenguaje “Markdown”; expliquemos un poco de que se trata:

Markdown

Markdown es un lenguaje de marcado ligero que se utiliza para dar formato a documentos de texto plano. Es muy útil para escribir contenido en línea como foros, blogs o documentos colaborativos porque permite dar formato al texto de una manera sencilla y rápida.

Algunos de los elementos que se pueden utilizar en markdown son:

  • Encabezados: se utilizan para organizar el contenido en secciones.
  • Listas: se pueden crear listas ordenadas o desordenadas.
  • Enlaces: se pueden incluir enlaces a otras páginas web.
  • Imágenes: se pueden insertar imágenes en el documento.
  • Énfasis: se puede agregar énfasis al texto utilizando negrita o cursiva.
  • Bloques de código: se pueden incluir bloques de código para resaltar el código fuente.

El formato de markdown es fácil de aprender y se puede utilizar en muchas plataformas diferentes. Además, muchos editores de texto y plataformas en línea tienen soporte nativo para markdown, lo que hace que sea muy fácil de usar.

Por ejemplo, para crear un encabezado en markdown, se puede utilizar el símbolo "#" seguido del texto del encabezado. Por ejemplo:

# Encabezado de nivel 1
## Encabezado de nivel 2
### Encabezado de nivel 3

Entender la Metodología de evaluación en Bug Bounty

De acuerdo con la metodología de pruebas WEB de OWASP, existen 2 categorías de pruebas:

Pruebas pasivas:

Las pruebas pasivas son aquellas que se realizan sin interactuar directamente con la aplicación web, y se realizan solo observando el comportamiento y los datos que se intercambian entre el cliente y el servidor. Estas pruebas sirven para recopilar información sobre la aplicación web, como su estructura, sus tecnologías, sus funcionalidades, sus vulnerabilidades potenciales, etc. Algunos ejemplos de pruebas pasivas son:

Análisis de tráfico web:

Consiste en capturar y examinar las peticiones y respuestas HTTP/HTTPS que se envían entre el navegador y la aplicación web, usando herramientas como Wireshark o Burp Suite. Esto permite identificar parámetros, cookies, cabeceras, métodos, errores, etc. que puedan revelar información sensible o insegura sobre la aplicación web.

Análisis de contenido web:

Consiste en analizar el código fuente HTML, CSS, JavaScript, etc. de las páginas web que se visitan, usando herramientas como el inspector de elementos del navegador; esto permite identificar comentarios, metadatos, scripts externos, formularios ocultos, campos de entrada, etc. que puedan revelar información sensible o insegura sobre la aplicación web.

Palabras que podrian ser útiles para un Bounty Hunter, de ser encontradas dentro de una página WEB:

userusernamesqlkey
hiddenclavecontraseñacmd
passwordusuariodbos
fieldodbcpasswdcampo

Análisis de robots.txt y sitemap.xml:

Consiste en buscar y examinar los archivos “robots.txt” y “sitemap.xml” que se encuentran en el directorio raíz de la aplicación web, usando herramientas como el navegador o la herramienta “curl”. Estos archivos indican qué páginas o recursos son accesibles o no para los motores de búsqueda, lo que puede revelar información sensible o insegura sobre la aplicación web.

Análisis de certificados SSL/TLS:

Consiste en verificar la validez y la seguridad de los certificados SSL/TLS que se usan para cifrar la comunicación entre el cliente y el servidor, usando herramientas como el navegador o SSL Labs. Esto permite identificar si la aplicación web usa un protocolo seguro y actualizado, y si el certificado es confiable y no está caducado o revocado.

Las pruebas pasivas son una parte importante de la metodología de pruebas web de OWASP porque permiten obtener una visión general de la aplicación web y sus posibles debilidades sin alterar su funcionamiento ni generar alertas en los sistemas de detección de intrusiones.

Pruebas Activas:

Las pruebas activas son aquellas que se realizan interactuando directamente con la aplicación web, enviando datos o comandos específicos para provocar y observar las respuestas del servidor. Estas pruebas sirven para verificar la existencia y el impacto de las vulnerabilidades potenciales que se identificaron en las pruebas pasivas, así como para descubrir otras que no se detectaron previamente.

Pruebas de Inyección y Fuzzing:

Consisten en enviar datos maliciosos a la aplicación web para intentar ejecutar comandos o consultas en el servidor o en la base de datos, usando herramientas como sqlmap o ZAP. Esto permite identificar si la aplicación web es vulnerable a ataques de inyección SQL, XSS, LDAP, etc.

Fuzzing en Sitios Web

El Fuzzing consiste en enviar datos aleatorios, inválidos y no esperados mediante los formularios de entrada de una aplicación buscando errores, comportamientos inesperados o alguna vulnerabilidad en ella. Un web fuzzer es un tipo de herramienta que permite probar qué rutas están activas y cuáles no en un sitio web, probando direcciones URL aleatorias y enviándoles parametros o pruebas para ver si estas funcionan y verificar cómo responden.

Un ejemplo de prueba de fuzzing en un sitio web sería enviar diferentes valores en un campo de entrada, como un formulario de inicio de sesión, para ver si la aplicación responde de manera inesperada o si devuelve un error. Por ejemplo, se podría enviar una cadena de texto mal formada en lugar de un nombre de usuario, para ver si la aplicación es vulnerable a inyección de código.

Pruebas de autenticación y autorización:

Consisten en probar la robustez y la seguridad de los mecanismos de identificación, autenticación y autorización de la aplicación web, usando herramientas como Hydra o Burp Suite. Esto permite identificar si la aplicación web es vulnerable a ataques de fuerza bruta, robo de credenciales, secuestro de sesiones, escalada de privilegios, etc.

Los ataques contra claves en sitios WEB más comunes son:

  • Ataque de fuerza bruta: Este ataque consiste en probar cada posible combinación de caracteres hasta que se encuentre la contraseña correcta. Puede ser una tarea que consume tiempo y requiere mucha potencia de procesamiento, pero puede ser efectiva si la contraseña es corta y simple.
  • Ataque de diccionario: Este ataque implica el uso de una lista de palabras o frases comúnmente utilizadas como posibles contraseñas. El atacante puede usar software para automatizar el proceso de probar cada palabra o frase hasta que se encuentre la contraseña correcta.

Si hablamos de Hacking, existen más ataques como por ejemplo:

  • Shoulder surfing: Esto implica observar al usuario mientras ingresa su contraseña, ya sea en persona o a través de una grabación de video. Puede ser un ataque efectivo si el usuario no es cuidadoso al proteger su contraseña.
  • Keylogger: Este es un tipo de malware que monitorea las pulsaciones de teclas del usuario, incluyendo las contraseñas, y envía la información de vuelta al atacante. Puede ser difícil de detectar y puede capturar contraseñas incluso si son complejas y únicas.

Pero en Bug Bounty nos interesan los dos primeros: Fuerza Bruta y Diccionario.

¿De qué lado vamos a explorar / explotar ?

Es muy importante tener en cuenta que no todas las vulnerabilidades que vamos a probar se encuentran del mismo lado, es decir, en algunos casos el lado vulnerable puede ser el Servidor de las aplicaciones WEB y en otros casos las vulnerabilidades se encuentran en el lado del Cliente (Usuario) que se conecta a un servicio específico.

Dado lo anterior, me parece prudente ofrecer este par de listas:

Vulnerabilidades del Lado del Servidor

Las vulnerabilidades del lado del servidor son aquellas que permiten a un atacante acceder a información o funcionalidades que no debería tener acceso en un servidor. Estas vulnerabilidades pueden permitir a un atacante ejecutar código malicioso en el servidor o acceder a datos confidenciales, como contraseñas de usuarios o información de tarjetas de crédito.

Algunos ejemplos de vulnerabilidades del lado del servidor incluyen:

VulnerabilidadDescripción
Inyección SQLPermite a un atacante enviar comandos maliciosos a una base de datos para obtener información confidencial o incluso modificar los datos almacenados en la base de datos.
Inyección de código del lado del servidor (SSI)Permite a un atacante ejecutar código malicioso en el servidor de una aplicación web. Esto se logra a través de la inserción de código malicioso en las solicitudes HTTP que la aplicación web utiliza para interactuar con el servidor. El código malicioso se ejecuta en el servidor, lo que permite al atacante controlar el comportamiento de la aplicación web y acceder a información confidencial.
Inyección de ComandosAdemás de la inyección SQL, existen otros tipos de inyección, como la inyección de comandos de sistema operativo
Server-side Request Forgery (SSRF)Permite a un atacante engañar a un servidor para que haga solicitudes a sistemas externos que no debería tener acceso. Básicamente, el atacante puede enviar una solicitud maliciosa al servidor que contiene una URL que apunta a un sistema externo controlado por el atacante. Si el servidor procesa la solicitud y se conecta al sistema externo, el atacante puede obtener información confidencial o incluso controlar el sistema objetivo.
Desbordamiento de búfer (Buffer Overflow)Permite a un atacante sobrescribir áreas de memoria RAM que no debería poder modificar, lo que puede permitir la ejecución de código malicioso.
Autenticación y autorización incorrectas (Rotas)Si el servidor no usa una autenticación y autorización adecuadas, un atacante puede acceder a información confidencial o realizar acciones que no debería poder realizar.
Fallas CriptográficasSon vulnerabilidades en los sistemas de cifrado que permiten a un atacante acceder a información sensible que debería estar protegida. Por ejemplo, si un sitio web utiliza un cifrado débil para proteger las contraseñas de los usuarios, un atacante podría interceptar las contraseñas y descifrarlas fácilmente. Esto puede permitir al atacante acceder a información confidencial, como la información financiera o personal de los usuarios.
Diseño inseguroSe refiere a la creación de una aplicación sin tener en cuenta aspectos de seguridad. Esto puede llevar a una arquitectura, lógica o funcionalidad defectuosas que son vulnerables a los ataques. Por ejemplo, una aplicación que almacena las contraseñas de los usuarios en texto plano es vulnerable a la extracción de contraseñas por parte de un atacante.
Error de configuración de seguridadOcurre cuando una aplicación o servidor no está configurado de manera segura, lo que lo hace vulnerable a ataques. Esto puede suceder cuando se usan configuraciones predeterminadas, servicios innecesarios o permisos mal configurados.

Vulnerabilidades del Lado del Cliente

Las vulnerabilidades del lado del cliente son aquellas que permiten a un atacante ejecutar código malicioso en el navegador del usuario o robar información del usuario. Estas vulnerabilidades pueden ser explotadas cuando un usuario visita un sitio web.

Algunos ejemplos de vulnerabilidades del lado del cliente incluyen:

VulnerabilidadDescripción
Control de acceso roto del lado del clienteControl insuficiente del acceso de JavaScript a los recursos (datos y código) del lado del cliente, filtración de datos sensibles o manipulación del DOM con fines maliciosos (para acceder a esos recursos). Al igual que OWASP Top 10: A01-2021 - Control de acceso roto, pero enfocado en el código del lado del cliente.
XSS basado en el DOMVulnerabilidades que permiten ataques XSS a través de la manipulación o el abuso del DOM.
Fuga de datos sensiblesIncapacidad para detectar/prevenir rastreadores y píxeles digitales en una propiedad web para garantizar el cumplimiento de las leyes de privacidad nacionales e internacionales.
Componentes vulnerables y anticuadosFalta de detección y actualizaciones de bibliotecas de JavaScript que están anticuadas o contienen vulnerabilidades conocidas. Al igual que OWASP Top 10: A06-2021 - Componentes vulnerables y anticuados, pero enfocado en bibliotecas del lado del cliente.
Falta de control de origen de tercerosEl control de origen permite la restricción de ciertos recursos o activos web al comparar el origen del recurso con el origen de la biblioteca de terceros. Sin aprovechar dichos controles, el riesgo de la cadena de suministro aumenta debido a la inclusión de código de terceros desconocido o no controlado que tiene acceso a datos en el origen del sitio.
Deriva de JavaScript (JavaScript Drift)Incapacidad para detectar cambios en el nivel de activos y código de JavaScript utilizado en el lado del cliente. Esto incluye la incapacidad para detectar cambios de comportamiento de este código para determinar si los cambios son potencialmente maliciosos en la naturaleza. Esto es particularmente importante para las bibliotecas de terceros.
Almacenamiento de datos sensibles en el lado del clienteAlmacenamiento de datos sensibles como contraseñas, secretos criptográficos, tokens de API o datos PII en almacenamiento persistente del lado del cliente como LocalStorage, caché del navegador o almacenamiento transitorio como variables de JavaScript en una capa de datos.
Fallos de registro y supervisión de seguridad del lado del clienteMonitorización y detección insuficientes de cambios y accesos de datos del lado del cliente, especialmente fallos y errores, en tiempo real mientras se ensambla y ejecuta cada página utilizando código de primera y tercera parte. Al igual que OWASP Top 10: A09-2021 - Fallos de registro y supervisión de seguridad, pero enfocado en el comportamiento del lado del cliente.

URLs para practicar

Es importante tener sitios web para practicar la detección de vulnerabilidades de forma legal por varias razones. En primer lugar, permite a los investigadores y pentesters practicar sus habilidades y mejorar sus técnicas de detección de vulnerabilidades en un entorno seguro y controlado. Esto puede ayudarles a prepararse para futuros trabajos o proyectos de seguridad.

Además, practicar en sitios web legales permite a los investigadores y pentesters trabajar dentro de los límites legales y éticos. Esto significa que no corren el riesgo de enfrentar consecuencias legales por sus acciones y pueden centrarse en encontrar y reportar vulnerabilidades de manera efectiva.

También es importante tener en cuenta que los sitios web para practicar la detección de vulnerabilidades suelen estar diseñados específicamente para este propósito. Esto significa que pueden contener una amplia gama de vulnerabilidades comunes y avanzadas que son difíciles de encontrar en sitios web reales. Esto permite a los investigadores y pentesters mejorar sus habilidades de detección de vulnerabilidades en un entorno controlado y seguro.
